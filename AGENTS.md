# vnpy_test 项目约束

## 沟通规则

- 与用户沟通时，默认使用中文。
- 解释尽量务实、易懂，适合 `vn.py` 学习场景。
- 以后默认使用中英双语 `git commit message`，通常先写中文，再写对应英文。

## 项目背景

- 这是一个用于学习 `vn.py` 的本地项目，默认按 A 股现货场景处理。
- 默认 Python 版本为 `3.13`。
- 默认虚拟环境为仓库内的 `.venv`。
- 除非用户明确要求切换，否则默认数据栈使用 `sqlite` 和 `akshare`。

## 策略规则

- 新增学习策略时，优先提供 A 股只做多版本。
- A 股示例默认按现货交易处理，不做空，除非用户明确要求。
- A 股示例参数在合理情况下优先按一手 `100` 股设计。

## 回测默认规则

- A 股入门回测优先使用 `601869.SSE` 和 `d` 周期。
- A 股现货回测默认参数优先使用：
  - `rate = 0.0005`
  - `slippage = 0.01`
  - `size = 1`
  - `pricetick = 0.01`
  - `capital = 100000`
- 如果本地数据库历史数据为空，需要先说明 `CTA回测` 读取的是本地数据库，再下载数据，不要先怀疑策略逻辑。

## 代码规则

- 以后新增或修改代码时，需要补充简单中文注释，帮助用户理解关键逻辑。
- 中文注释应简短清楚，优先解释关键计算、业务判断、流程转换。
- 不要为一眼就能看懂的简单赋值或导入写注释，避免无意义注释过多。

## 运行规则

- 启动项目时，使用 `.venv/bin/python run_vnpy.py`。
- 修改 GUI 或启动逻辑时，注意兼容 macOS，并确认启动路径仍然可用。
- 每次修改完代码后，如需让改动立即在界面中生效，应使用 `scripts/restart_vnpy_gui.sh` 强制清理所有旧的 `run_vnpy.py` 实例，再启动新的 GUI，并明确告知用户已经完成重启。
- 不要悄悄修改用户级 `~/.vntrader` 配置；如果必须改机器相关配置，要明确告知用户。

## 记录规则

- 仓库内维护 [daily_work.md](/Users/zezhang/Documents/codex/vnpy_test/daily_work.md) 作为开发工作日志。
- 每次重要 Git 提交后，都应按“具体时间、提交号、提交信息、详细说明”的格式追加记录，方便后续回顾开发过程。
- 如同一次任务既包含代码/文档改动，又需要补记 `daily_work.md`，默认合并到同一个 Git 提交中完成，不再拆成两个独立提交。

## 安全规则

- 推送远端前先询问用户。
- Git 提交信息默认使用中英双语，通常先写中文，再写对应英文。
- 安装 Homebrew 包、修改全局 Shell 配置等仓库外系统级改动前先询问用户。
